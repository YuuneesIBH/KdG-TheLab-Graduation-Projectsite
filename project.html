◈ PROJECT DNA

This project does not start from the idea of a simple launcher. It starts from the idea that a digital play environment can also be a physical, technical, and interactive installation.

One side delivers a complete cabinet flow with boot screens, CRT effects, local games, and gamepad controls. The other opens Guppy, a lab-oriented environment where the desktop app works directly with firmware, tools, scans, and external modules.

◈ PROJECT ARTICLE
CENTRAL RESEARCH QUESTION
How can a single desktop platform combine the experience of a classic arcade cabinet with a flexible hardware- and security-oriented working environment without making the interface fragmented or unclear for the user?

Introduction

The Arcaders was developed as a graduation project within The Lab and starts from a clear tension. An arcade environment depends on atmosphere, simplicity, and rhythm, while a technical lab environment requires control, feedback, and extensibility. Many digital projects therefore choose one direction or the other. This project investigates whether both worlds can be brought together in a credible way inside one fullscreen desktop application.

The main question is how one application can function both as an accessible arcade experience and as a practical hardware interface. Two supporting questions are central here: how can users move intuitively between both modes, and what architecture is needed to connect software, firmware, and local tooling in a safe and understandable way?

A Clear Two-Mode Structure

The answer to that research question starts with the structure of the application. Instead of placing every feature inside one screen, the platform is intentionally built around two clearly defined modes: Arcade Mode and Guppy Mode. This separation prevents visual noise and immediately clarifies the role of each environment for the user.

Arcade Mode focuses on experience and accessibility. Guppy Mode focuses on hardware, status information, and tooling. The two modes therefore have different purposes, but they remain visually connected through one shared design language. That balance between separation and consistency is what makes the platform coherent. Users experience it as one product without feeling that incompatible goals are competing inside the same interface.

Arcade Mode as an Accessible Entry Point

The arcade side demonstrates that usability does not need to conflict with technical diversity. The launcher brings together eleven local games inside one visual shell, even when those titles run on different runtimes such as Python, Java, or native executables. For the user, this does not feel like a collection of unrelated programs, but like one platform with a consistent identity.

That matters because the first experience determines whether the project is convincing during a demo or conference setting. Through a fullscreen presentation, CRT-inspired effects, glitch animation, and gamepad-friendly navigation, the platform creates a readable and rhythmic flow. Arcade Mode therefore works as a low-threshold entry point into a project that is technically more complex than a traditional game launcher.

Guppy Mode as the Technical Second Layer

The second layer of the project is Guppy Mode, a lab-oriented environment that communicates over USB serial with a self-built ESP32 board (Espressif Systems, n.d.). This mode shows that the same desktop application can also function as a useful environment for hardware interaction, live tooling, and technical demonstration workflows. The application detects the board, reads the capabilities reported by the firmware, and adapts the interface to what the connected device actually supports.

That device-aware approach is a direct answer to the research question. It prevents users from seeing functions that are not available at that moment and makes the technical layer more concrete and reliable. Examples of functionality in Guppy include IR commands, Wi-Fi scans, audit information, and access-point profile management. At the same time, the context remains clearly limited: these security-oriented features are intended only for controlled and authorized demo or learning environments.

Architecture and Local Intelligence

The connection between both modes is supported by a layered architecture. Electron provides the desktop shell and process control (Electron, n.d.), React handles the interface and state transitions (React Team, n.d.), and TypeScript supports the technical structure of the application (Microsoft, n.d.). A preload layer separates the user interface from system-level actions, allowing local games, serial communication, storage, and external processes to be managed centrally without making the interface unnecessarily complex.

An additional strength is the local hint layer powered by Ollama (Ollama, n.d.). During gameplay, the application can generate short contextual hints based on the selected game, genre, and difficulty. This feature is intentionally supportive rather than dominant. It improves usability without interrupting the arcade flow and also demonstrates how local AI can be integrated into a desktop project in a practical way.

Conclusion

This project shows that combining an arcade experience with a hardware-oriented lab environment is feasible, provided that the application is not treated as a loose collection of tools but as one clear user experience with separated modes, a consistent visual language, and a device-aware architecture. The central research question can therefore be answered positively: one desktop platform can unite both worlds without losing clarity, as long as structure, role separation, and context remain central in the design.

The Arcaders therefore positions itself not only as a demonstration of frontend development, desktop orchestration, and embedded integration, but also as a grounded case study of how playfulness and technical functionality can strengthen each other within one coherent product.

◈ HINT LAYER & EXTENSIBILITY

The hint layer is deliberately supportive rather than dominant. During play, the platform can generate short contextual hints that make classic software more approachable without breaking the cabinet rhythm.

The architecture is also open enough to grow with the project. Additional guidance flows, synchronized sessions, or connected services can be added later without changing the core split between arcade and Guppy.

CURRENT USE
Local hints through Ollama based on game context and difficulty.
EXTENSIBLE SETUP
The same interface can later support richer guidance or connected services if the project grows beyond the local demo.
NEXT STEPS
Future multiplayer flows and synchronized demo moments fit naturally into the current structure.
SAFE CONTEXT
Security-oriented features remain limited to controlled and isolated learning environments.
◈ CONTEXT & VALUE
01
The security and network-oriented features are intended for controlled, authorized demo and lab environments. The project is explicitly not designed for use outside a permitted context.
02
What makes the project strong is the combination of disciplines: frontend, desktop orchestration, embedded firmware, physical interaction, and a local hint layer meet in one tangible installation.
03
The current version already demonstrates a working combination of launcher, local games, hardware bridge, and hint layer. It feels playful in form, but solid in structure.
04
In the longer-term vision, the platform can evolve toward a broader arcade ecosystem with synchronized sessions, richer demo tooling, and room for new cabinet experiences.
